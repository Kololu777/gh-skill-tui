現在の結論： gh-skill-check の合否は tree SHA ではなく、選択した source revision と インストール済みファイルの実質的な内容一致で決まる。SHA は由来・高速な更新候補表示に使えるが、 checker の正しさの根拠にはしない。一方で TUI と gh skill update は、まだ SHA を状態表示に使う。

状態を構成する5つの軸

「installed / outdated」の一列だけで表すと、信頼性・配置・同期状態が混ざる。 現行実装を理解するには次の軸を独立に考えるのがよい。

01

由来 / provenance

github-repo または local-path が許可元に属するか。結果は managed / foreign / untracked。

02

configured source 所属

許可済みであるだけでなく、いま開いている source と同一か。別の許可リポジトリもここでは outside。

03

配置 / coverage

どの physical destination に存在するか。複数agentが同じ .agents/skills を共有する。

04

内容 / content

source revision のファイル集合と同じか。checkerではこの軸が合否の正本。追加・変更・削除を比較する。

05

追跡メタデータ / tracking

github-tree-sha、tui-tree-sha、refなど。欠落・古さと内容不一致は別問題として扱う。

source snapshot と installed snapshot を別々に作り、最後に同じ skill key で結び付ける。

Source snapshot

GitHub tree API、または local Git の tracked tree。skill path、tree SHA、各fileのblob SHA、SKILL.md本文。

Canonical comparison

source keyで対応付け。補助fileはblob、SKILL.mdは注入metadataを除いた意味比較。

←

Installed snapshot

各skills rootを走査。frontmatter metadata、raw SKILL.md、file集合とcontent hashを保持。

Remote source

  • pin があれば pin、なければ ref、どちらもなければdefault branchを選ぶ。
  • recursive tree APIからskill directoryのtree SHAと、全tracked fileのblob SHAを得る。
  • TUIはremote SKILL.md 本文を必要時にlazy loadする。
  • checkerは対象scopeで必要な SKILL.md 本文を先に読み、意味比較を完結させる。

Local source

  • git ls-files でtracked skillだけを発見し、tree/blob SHAは HEAD から得る。
  • SKILL.md 本文はworking treeから読むため、未commitのSKILL.md変更は意味比較に現れる。
  • 一方、補助fileの比較元blobはHEAD。未commitの補助file変更はsource snapshotへ反映されない。
  • local cloneがoriginより古いかどうかは、このツールの検査範囲外。

<root>/*/SKILL.md と <root>/*/*/SKILL.md。通常名と author/skill の2段namespaceに対応する。

状態をどう導出するか

同じinstalled copyでも、「許可元か」と「現在のconfigured sourceか」は別判定。

managed

GitHub repo slugがallowed sourcesに一致、またはlocal pathがallowed local roots配下。

foreign

GitHub metadataはあるが、repo slugがallowlistにない。由来は分かるが信頼対象外。

untracked

信頼できるsource metadataがない。手置き、自作、他tool由来などを含む。

TUIのremote SKILL.md 本文はlazy loadされるため、SHAが一致または欠落していて 差分がSKILL.mdだけにある場合、本文load前は一時的に ✓、load後に m となり得る。 checkerは判定前に必要な本文をloadするので、この一時状態はない。

content-based promotion： outside copyとsource側の同名pathのfile集合が完全一致した場合、TUI/checkerはそのscan中だけmanaged相当に昇格できる。 import PRがmergeされた後を自動認識するための処理で、installed SKILL.md 自体は書き換えない。

「diffが合っている」の定義

単純なdirectory byte diffではない。installerが注入・再整形する差を取り除いてから比較する。

SKILL.md

  • github-path、github-ref、github-repo、github-tree-sha を除去。
  • github-pinned、local-path、tui-tree-sha も除去。
  • 空になった metadata: mappingを除去。
  • frontmatterのtop-level順序、child順序、indent差をcanonicalize。
  • installerが除くfrontmatter直後の空行や末尾改行差を無視。
  • 本文やユーザー独自metadataの意味のある変更は差分。

Companion files

  • 各file内容をGit blob SHA-1形式でhashし、source treeのblob SHAと比較。
  • sourceにあるのにinstalledにないfileは削除差分。
  • installedにだけあるfileは追加差分。
  • file mode / executable bitは比較しない。内容のみ。
  • __pycache__、node_modules、.DS_Store は両側で無視。
sourceFilesFor(skillKey)  = { relative/path -> source blob SHA }
installed.FileShas       = { relative/path -> installed blob SHA }

equal = same file set
     && same companion-file blobs
     && skillMDEquivalent(source.SKILL.md, installed.SKILL.md)

gh-skill-check の判定手順

CI向けのread-only audit。更新・PR作成は行わない。

  1. source revisionを解決remote treeまたはlocal HEADからskill、tree、blob一覧を作る。
  2. 全targetをscanuser/projectのphysical rootをdedupeしてinstalled snapshotを作る。
  3. configured sourceをauthoritativeにするcheckerではproject sourceをallowlistへ加え、比較対象として認識する。
  4. 対象scopeのSKILL.mdをloadremote sourceでも意味比較できるよう、必要なdistinct skill本文を取得する。
  5. ignoreを先に適用name、github path、basenameにglob patternを当てる。outsideにも適用。
  6. provenanceとsource pathを確認別sourceはoutside、sourceにpathがなければmissing-source。
  7. contentを比較tree SHAは見ず、file集合・companion blob・semantic SKILL.mdが一致すればPASS。
  8. 終了codeを返す問題なし=0、audit issue=1、設定/API/実行error=2。

検出するもの

  • configured sourceに属さないoutside copy
  • sourceから消えたtracked path
  • installed fileの追加・変更・削除
  • scan不能target
  • 不正なignore globやsource取得失敗

検出しないもの

  • source skillが未installであること
  • 選択agentすべてへの必須coverage
  • tree SHA metadataの欠落・古さ
  • 無視対象cache/litterの差
  • file mode差、local cloneのorigin遅延

状態が変わるとき

A

Remote install / update

gh skill install がrepo/path/ref/tree SHA metadataをSKILL.mdへ注入。次回scanでsource keyと結び付く。

B

Local install / update

GitHub CLIが local-path を注入し、TUIが成功後に tui-tree-sha を追記する。

C

Local edit

FileShasとsource blobsがずれる。TUIでは m、checkerでは modified。TUIのpはupstream PRを組み立てる。

D

Outside import / adopt

outside copyはpでsourceへimport。merge後に内容一致すればmemory上でmanagedへ昇格。source rowのinstallは既存outsideをadopt/replaceする。

E

Rescan

install/delete/PR操作後、または r でinstalled rootsを再走査する。 状態を保存する独立databaseやlockfileはなく、source snapshot + filesystem + SKILL.md metadataから都度再構築する。

現在の非対称性と整理候補

ここは「実装済み」と「次に揃えるなら」を分けて読む。

1. checkerはcontent優先、TUIはtree SHA優先
同じ内容でもtracking SHAだけ古い場合、checkerはPASS、TUIは ↓ outdated。 一貫させるならTUIもcontentを正本にし、metadata staleを別badgeにするのが分かりやすい。
2. 「modified」は差分の方向を断定できない
内容差分は「installedを手で編集した」「sourceが進んだ」「両方」のどれでも起きる。 checkerのissue名を content-drift にすると意味がより正確になる。
3. repair hintがphysical destinationを持っていない
現在のchecker actionはskill pathとscope中心で、.claude の --agent や .opencode の --dir を完全には表現しない。issueへtarget情報を渡せば正確なfixを生成できる。
4. checkerはcoverage policyではない
既存copyの健全性は検査するが、「この5skillをClaude/Codex/OpenCodeへ必ず配置する」は表せない。 必要なら required_skills / required_agents のような別policyが必要。
5. local source snapshotが一部working tree、一部HEAD
SKILL.md本文はworking tree、companion blobはHEAD。sourceを完全にHEAD基準にするか、 完全にworking tree基準にするかを明示的に選ぶ余地がある。
6. remote semantic checkはskillごとの本文取得が必要
recursive tree一回だけではfrontmatterの意味比較ができないため、checkerは対象distinct skillの本文を追加取得する。 正確さと引き換えにAPI round tripが増える。

gh-skill-check --scope project
